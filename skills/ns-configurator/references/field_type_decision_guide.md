# Field Type Decision Guide

Choosing the wrong field type is a permanent mistake — after data is entered, the field type cannot be changed. Use this guide before creating any custom field.

---

## Quick Decision Table

| Data to Store | Field Type | Notes |
|--------------|-----------|-------|
| Short text (name, code, reference) | **Free-Form Text** | Max 300 chars. Use for IDs, names, codes. |
| Long text (notes, descriptions) | **Long Text** (Text Area) | No character limit. Not searchable with `is`. |
| Whole number | **Integer Number** | No decimal places. |
| Money / currency amount | **Currency** | Applies the account's currency formatting. |
| Decimal number (rate, percentage) | **Decimal Number** | Specify decimal places. |
| Percentage | **Percent** | Stores as decimal (0.25 = 25%); formats as % in UI. |
| Yes / No flag | **Checkbox** | Default: unchecked. Use for boolean flags. |
| Date only | **Date** | No time component. Use for due dates, effective dates. |
| Date + Time | **DateTime** | Use for audit timestamps, event times. |
| Fixed list of values | **List/Record → Custom List** | Create a `customlist_` first, then source it. |
| Another NetSuite record (dynamic) | **List/Record → [Record Type]** | Links to and sources from a specific record type. |
| Email address | **Email Address** | Renders as a mailto: link in the UI. |
| Phone number | **Phone Number** | Formatted display; no validation by default. |
| URL / hyperlink | **URL** | Renders as a clickable link. |
| File attachment | **Document** | Links to a file in the File Cabinet. |
| Rich text / HTML content | **Rich Text** | Only for user-facing formatted content; avoid for data. |
| Multiple selections from a list | **Multiple Select** | Use sparingly — difficult to filter on in searches. |
| Inline HTML display | **Inline HTML** | Read-only; generated by script. |
| Image | **Image** | Links to an image file. |

## Detailed Field Type Rules

### Free-Form Text
- **Use when:** Short identifiers, reference codes, names under ~100 characters
- **Don't use when:** The value should come from a controlled list (use List/Record instead) or the text could be multi-line (use Long Text)
- **Filter compatibility:** Supports `contains`, `is`, `starts with`, `is not empty`

### List/Record
- **Use when:** The field value must come from a predefined set of options (status, category, type) OR must link to another NetSuite record (Customer, Employee, Item)
- **Source options:**
  - `Custom List` — a fixed dropdown (e.g., Approval Status: Pending / Approved / Rejected)
  - `[Record Type]` — a dynamic lookup to a NetSuite record (e.g., link to Customer record)
- **Don't use when:** The list of values may grow unpredictably — consider a Custom Record with a search instead
- **Filter compatibility:** Supports `is`, `is not`, `any of`, `none of`

### Checkbox
- **Use when:** A simple true/false flag (feature enabled, record processed, approval required)
- **Default:** Always set the default explicitly — unchecked is not always the right default
- **Don't use when:** There are more than 2 states — use List/Record instead
- **Filter compatibility:** `is true`, `is false`, `is not empty`

### Currency vs. Decimal Number
- **Currency:** Use for monetary amounts that should display with the currency symbol and respect multi-currency settings
- **Decimal Number:** Use for non-monetary decimals (weight, dimensions, percentages not in the Percent type, ratios)
- **Do not** use Integer for money, even if the amounts are always whole numbers

### Date vs. DateTime
- **Date:** Use for business dates (due date, ship date, effective date) where time is irrelevant
- **DateTime:** Use for system timestamps (created date, processed at, last synced) where time matters for ordering or elapsed-time calculations
- **Warning:** DateTime values are stored in UTC; display depends on the user's timezone setting

### Long Text
- **Use when:** Notes, descriptions, comments, or any text that may exceed one line
- **Limitation:** Cannot use `contains` in saved search filters efficiently on very long text; prefer shorter Free-Form Text fields for data you need to search on
- **Don't use for:** Structured data that should be parsed — use separate fields

### Multiple Select
- **Use sparingly:** Multiple Select fields are stored as a comma-separated string, making them difficult to filter precisely and impossible to join in saved searches
- **Alternative:** Consider a child custom record with a List/Record field and one row per selection — this is more flexible and searchable

## Custom Record vs. Custom Field Decision

Before adding another field to a transaction or entity record, ask:

| Situation | Decision |
|-----------|----------|
| One value per transaction (e.g., approval status) | Custom field on the transaction body |
| Multiple values per transaction (e.g., multiple approval events) | Custom record with a parent link field |
| One value per line (e.g., lead time per item line) | Custom column (line) field on the sublist |
| Standalone data not tied to a transaction | Custom record |
| Data that needs its own list view and search | Custom record |
| Data shared across multiple record types | Custom record with relationship fields |

## Field Application Strategy

Do not apply a field to **all record types** or **all forms** by default.

| Rule | Reason |
|------|--------|
| Apply to only the record type(s) where the data belongs | Fields applied to all records clutter unrelated forms and slow page loads |
| Apply to only the forms where the field is relevant | Users shouldn't see fields they can't or shouldn't fill in |
| Mark hidden fields on forms as "hidden" not absent | Scripts can read fields that aren't on the form; absent fields don't appear in sublist columns |
| "Store Value" on sourced fields | If a field sources its value from a related record, enable Store Value so the value persists if the related record changes |
